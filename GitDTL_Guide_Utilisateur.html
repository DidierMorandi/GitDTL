GitDTL - Guide utilisateur

Guide pratique pour gérer au quotidien un projet Git avec l'interface GitDTL.

GitTkinterWindowsv1.0-3

Objectif

GitDTL est une interface Windows qui rend les opérations Git courantes accessibles depuis un menu numéroté. L'application lance les commandes Git dans le dossier du projet choisi, affiche les résultats et conserve une trace dans un journal local.

Le principe est simple : regarder l'état du projet, enregistrer les fichiers utiles, valider les changements, puis publier sur GitHub lorsque le projet est prêt.

Démarrage et choix du projet

  1. Ouvrez GitDTL.
  2. Vérifiez le dossier affiché en haut de l'interface.
  3. Utilisez Changer de projet si ce dossier n'est pas le bon.
  4. Si le dossier n'est pas encore un dépôt Git, acceptez l'initialisation proposée avec git init.
GitDTL peut créer un dépôt Git local, mais il ne crée pas automatiquement le dépôt correspondant sur le site GitHub.

Routine Git conseillée

1. Lire l'état

Commencez par l'option 1 pour savoir si des fichiers sont modifiés, nouveaux, déjà enregistrés ou prêts à être publiés.

2. Voir les modifications

Utilisez l'option 2 avant de valider. Elle permet de relire les différences et d'éviter d'inclure une modification accidentelle.

3. Enregistrer puis valider

Ajoutez les fichiers pertinents avec l'option 4, puis créez un commit avec l'option 6 et un message court mais explicite.

4. Publier

Après un commit, l'option 7 envoie les changements vers le dépôt distant configuré sur GitHub.

Publier sur GitHub

L'option 7 publie les commits locaux vers GitHub. Avant d'envoyer les données, GitDTL vérifie si le projet contient encore des fichiers non validés, si le remote origin existe et si la branche locale possède une branche GitHub associée.

  • Si aucun remote n'existe, GitDTL demande l'adresse du dépôt GitHub.
  • Si la branche n'a pas encore d'upstream, GitDTL peut lancer git push --set-upstream origin <branche>.
  • Si des fichiers ne sont pas validés, GitDTL les liste et demande confirmation avant de publier quand même.

Créer une version

L'option 8 sert à marquer officiellement une étape importante du projet. GitDTL demande un numéro de version, ajoute le préfixe v au tag, puis enchaîne les commandes nécessaires.

git commit -m "Version 1.0.0"
git tag -a v1.0.0 -m "Version 1.0.0"
git push
git push origin v1.0.0
Cette action crée et publie un tag Git, mais elle ne crée pas automatiquement une Release GitHub. La Release doit être créée manuellement sur GitHub si elle est nécessaire.

Journal, aide et diagnostic

GitDTL écrit les actions importantes dans logs\gitdtl.log. L'option 12 permet de relire ce journal, de l'actualiser, de l'effacer après confirmation ou d'en exporter une copie.

Les boîtes de dialogue peuvent afficher une aide contextuelle issue de aide.md. Les messages Git inattendus peuvent être enrichis par des conseils définis dans expert_git.md.

En cas de problème

  1. Lancez l'option 1 pour lire l'état du projet.
  2. Lancez l'option 11 pour obtenir un diagnostic technique.
  3. Ouvrez l'option 12 pour consulter le journal.
  4. Vérifiez que Git est installé et disponible dans le PATH Windows.
  5. Vérifiez le dossier projet affiché avant toute action sensible.
Le scan Git du matin, lorsqu'il est installé, reste en lecture seule : il consulte l'état local des dépôts sans créer de commit, sans modifier les fichiers et sans publier sur GitHub.
